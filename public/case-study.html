System Design Case Study

A production-grade fintech product operated by a single engineer

Single-Person Engineering Operation

A full-stack financial technology product — mobile app, backend API, data enrichment pipeline, AI integrations, and infrastructure — designed, built, and operated by one person. Every architectural decision was made with the constraint of solo operation in mind: high automation, minimal manual intervention, strong security defaults, and systems that explain themselves.

React Native / Expo Node.js / TypeScript PostgreSQL + Prisma Docker Compose GitHub Actions CI/CD Cloudflare CDN + WAF Tailscale VPN OpenAI GPT-4o-mini Plaid API Twilio Verify Stripe AES-256-GCM Encryption GDPR / CCPA Compliant

Infrastructure & Hosting

Self-hosted on a dedicated Linux server, with Cloudflare sitting in front for CDN, DDoS protection, and WAF. Admin access is gated through Tailscale — no public SSH exposure. Every service runs in Docker containers with explicit restart policies and health checks.

Cloudflare Setup

Cloudflare acts as the first layer — handling TLS at the edge, absorbing DDoS traffic, and enforcing WAF rules before a single packet reaches the server. A Cloudflare Origin Certificate is installed on the server for double-TLS termination, ensuring the connection is encrypted end-to-end even between Cloudflare and the origin.

Tailscale for Admin

No port 22 is publicly exposed. All administrative access — deploys, database inspection, log tailing, script execution — goes through a Tailscale VPN mesh. The server is only reachable by authenticated devices on the private network. This eliminates an entire category of attack surface.

Docker Architecture

  • Two containers: API (Node.js) and PostgreSQL 16
  • All ports bound to 127.0.0.1 — nothing exposed to the public network
  • Named volume for database persistence
  • Health checks on both containers (HTTP + pg_isready)
  • Multi-stage Dockerfile: builder compiles TypeScript, runner is lean prod image with non-root user

Git Workflow & CI/CD Pipeline

GitHub is the source of truth. Every push to master triggers a GitHub Actions pipeline with self-hosted runners installed directly on the staging and production servers. No third-party deployment service — the runner has direct access to run deploy scripts on the machine.

Self-Hosted Runners

GitHub Actions runners are installed on the actual servers — staging and production each have their own. This means deployments don't require exposing any ports or SSH keys to GitHub's infrastructure. The runner has the necessary local permissions to pull code, run Docker commands, and execute deploy scripts.

The deploy script runs prisma migrate deploy (safe, additive-only migrations in production), rebuilds the container, and waits for the health check to pass before marking the deployment complete.

Staging → Production Gate

Staging deploys automatically on every push to master. Production deployment requires a manual workflow_dispatch with an explicit confirmation input (deploy_prod: yes). This creates a hard human gate between staging validation and production release — a small but meaningful safety mechanism when operating alone.

The production compose file pulls a pre-built image by IMAGE_TAG from GitHub Container Registry (ghcr.io) rather than building on the server — keeping deploys fast and reproducible.

Security Architecture

Defense-in-depth across five distinct layers. Each layer assumes the previous one can fail. No single point of trust — sensitive data is encrypted at rest regardless of whether the database is accessed legitimately or not.

L1 Network Cloudflare DDoS + WAF, TLS 1.2/1.3 only, all ports localhost-bound
L2 HTTP Helmet.js headers (HSTS, CSP, X-Frame-Options), nginx rate limiting (10 req/s per IP, burst 20), CORS whitelist
L3 Auth Passwordless phone OTP via Twilio, JWT HS256 (30-day expiry), tokens stored as SHA-256 hash only — raw token never persisted
L4 Data AES-256-GCM encryption at rest for all sensitive fields: Plaid access tokens, merchant names, transaction data, phone numbers, IP addresses
L5 External Stripe webhook signature verification, Plaid read-only access scope, per-endpoint rate limits (OTP: 3/10min, sync: 10/5min)

Encryption at Rest

Every piece of sensitive data is encrypted with AES-256-GCM before it touches the database. This includes Plaid access tokens, transaction merchant names, the global merchant name cache, phone numbers stored in OTP sessions, and user IP addresses captured for consent records. The encryption key is a 64-character hex string validated at startup — the server refuses to start without it.

Compliance Framework

GDPR and CCPA compliance is built into the data model, not bolted on afterward. A Consent table records explicit consent at sign-up: timestamp, IP address, terms version, privacy version, data processing consent, and financial data access consent (Plaid). Full account deletion cascades cleanly through all tables and revokes Plaid access tokens. Consent is not optional — it's a precondition for auth.

Data Enrichment Pipeline

Raw transaction data from banks is noisy and inconsistent. The enrichment layer transforms it into structured, reliable, and increasingly accurate information. This is the core intelligence layer of the system — not the AI features visible to users, but the infrastructure that makes those features trustworthy.

The core insight: Plaid provides category labels, but they're coarse and inconsistent across institutions. A merchant appearing as "AMZN*" at one bank and "AMAZON.COM" at another needs to resolve to the same canonical entity. Enrichment is what makes that happen — and it compounds over time as more users interact with the same merchants.

Enrichment pipeline — from raw transaction to structured insight

1 — Ingest: Cursor-Based Bank Sync

Plaid's transaction API uses a cursor for incremental sync. Each sync call returns only new/modified/removed transactions since the last cursor position. Added and modified transactions enter the enrichment queue. Removed transactions are deleted. The cursor is persisted per PlaidItem, enabling safe resumption after failures without duplicating or missing data.

2 — Store: Encrypted at Write

Transaction merchantName is encrypted with AES-256-GCM at write time. All database reads decrypt on the way out. This means even direct database access — whether authorized or not — exposes only ciphertext. The encryption key lives only in the application environment, never in the database.

3 — Merchant Cache Lookup (Global)

The Merchant table is a shared, cross-user cache of known merchants. Merchant names are HMAC'd for O(1) lookup without exposing plaintext in the index. If a merchant is found with sufficient confidence, its enrichment data is applied directly — no AI call needed. The sourceCount field tracks how many users have encountered this merchant; higher counts increase confidence. lastVerifiedAt timestamps enable periodic re-validation.

4 — AI Classification (Cache Miss)

On a cache miss or low-confidence merchant, the transaction is submitted to OpenAI (GPT-4o-mini) for classification. The model returns: primary category, subcategory, transaction type (subscription, transfer, income, one-time purchase, etc.), domain, and a structured confidence score per field. Results also include flags — for example, RAISED is set when a recurring transaction amount has increased, signaling a price change to the user. The model version is stored with every enrichment record, enabling future re-classification as models improve.

5 — Enrichment Record + Merchant Update

A TransactionEnrichment record is created linking the transaction to its enrichment result. In parallel, the Merchant table is updated or created: sourceCount is incremented, confidenceScore is recalculated, and nameEncrypted stores the canonical display name. The next transaction from any user with the same merchant hits the cache directly — the AI call is amortized across the user base.

6 — User Correction Feedback Loop

Users can submit corrections to enrichment results via POST /enrichment/correct. A UserCorrection record captures the field corrected, the old value, and the new value, linked to both the enrichment record and the user. Corrections feed back into merchant confidence scoring — enough corrections on a merchant trigger re-evaluation. This creates a supervised feedback loop where the data improves continuously without requiring manual curation.

7 — Retry & Resilience

Failed enrichments (API errors, timeouts, unexpected model output) are not dropped. The TransactionEnrichment record tracks retryCount and nextRetryAt with exponential backoff. The enrichment source field (plaid, ai, manual) provides auditability — every enrichment has a traceable origin. Pending enrichments never block the transaction from being displayed; the UI degrades gracefully until enrichment resolves.

Why this matters architecturally: The enrichment pipeline is a write-optimized, eventually-consistent system with a shared global cache that improves with scale. New users benefit from enrichments already computed for previous users. The AI cost is incurred once per unique merchant, not once per transaction — making the system economically viable and faster over time. This is the same pattern used in production data enrichment at scale, implemented here as a solo project.

AI Integration

AI is used in three distinct ways: as a classification engine in the data pipeline, as an insight generator, and as a conversational interface. Each use case has different latency requirements, cost profiles, and failure modes — and each is handled differently.

Transaction Enrichment

GPT-4o-mini / Batch classification

AI classifies transactions into structured categories, detects transaction types, and flags anomalies like price increases. Results are cached in the merchant layer so the model is called once per unique merchant, not per transaction.

AI-Generated Insights

GPT-4o-mini / Rate-limited generation

Analyzes 90 days of enriched transaction data to surface specific insights: spending spikes, recurring subscriptions detected, savings opportunities. Free tier: 1/month. Pro tier: 4/month. Insights older than 60 days are auto-pruned.

Financial Score Explanation

GPT-4o-mini / Cached per month

The monthly financial score (savings-rate based, 0–100) is accompanied by an AI-generated plain-language explanation. The explanation is cached in the score record and regenerated only when the score changes — not on every request.

Conversational AI Chat

GPT-4o-mini / Pro feature

Users can ask questions about their finances in natural language. Context is assembled from enriched transaction history, score data, and subscription information before each message. Rate limited at 20/hour. Pro-tier only.

Lazy client initialization: The OpenAI client is instantiated on first use, not at server startup. This ensures that a missing or invalid API key doesn't crash the server — it surfaces as a runtime error only when an AI feature is actually called, keeping the rest of the API functional.

External Services

Four external APIs are integrated, each with its own authentication pattern, webhook or polling strategy, and failure handling. The architecture is designed so that a failure in any one service degrades gracefully without taking down unrelated functionality.

Plaid

Bank data connectivity

Read-only access to transaction history, account balances, routing/account numbers, and ML-detected recurring streams. OAuth-based Link flow with access tokens encrypted at rest. Cursor-based incremental sync prevents data duplication across sync calls.

Transactions API Recurring API Auth API

Twilio Verify

Passwordless authentication

Phone number OTP via SMS — no passwords stored anywhere in the system. Twilio Verify handles the code generation, delivery, and validation. OTP attempts are tracked in the DB with expiry timestamps. Rate limited at 3 requests per 10 minutes per IP.

SMS OTP E.164 validation

Stripe

Subscription billing

Checkout sessions for monthly and annual plans. Customer portal for self-serve subscription management. Webhook handler validates Stripe signatures before processing any event — subscription lifecycle events (created, updated, deleted, payment failed) all update the User record in real time.

Checkout Sessions Webhooks Customer Portal

Expo Push

Mobile push notifications

Expo's managed push notification service for React Native. Push tokens are registered at login and stored per user. Notifications are triggered on financial score changes and new AI insight generation — keeping users informed without requiring them to open the app.

Push Tokens Event-driven delivery

Mobile Application

React Native with Expo for cross-platform iOS and Android. File-based routing with expo-router, two-layer state management (Zustand for auth/global state, TanStack Query for server state with 48-hour cache), and biometric authentication via Expo's local auth module.

Over-the-Air Updates

Expo Updates is configured for OTA delivery — JavaScript bundle changes (UI fixes, logic updates, non-native changes) can be pushed directly to users' installed apps without an App Store review cycle. This is a significant operational advantage for a solo operator: bug fixes ship in minutes, not days.

State Architecture

Auth state (token, userId) lives in Zustand and is loaded from Expo SecureStore at startup — tokens are never in plain memory longer than necessary. Server data (transactions, scores, insights) goes through TanStack Query with a 5-minute stale time and 48-hour garbage collection. The query cache is persisted to AsyncStorage, so the app is usable immediately on reopen, even before network requests complete.
